# 기능 정리

## 1. 서비스 개요

`My News`는 카테고리 기반 뉴스 탐색, 익명 개인화 추천, 트렌딩 키워드 정리, 위치 기반 날씨 확인을 한 흐름 안에서 제공하는 개인 뉴스 서비스입니다.

이 서비스의 가장 중요한 기능은 `나의 맞춤 뉴스`입니다.  
로그인 없이도 사용자가 실제로 클릭하고 읽은 기사 흐름을 반영해, 다음 방문 때 더 관심 있을 만한 기사를 앞쪽에 배치하는 것이 핵심입니다.

## 2. 핵심 기능: 나의 맞춤 뉴스

### 2.1 기능 목적

`나의 맞춤 뉴스`는 단순히 최신 기사를 시간순으로 보여주는 기능이 아닙니다.  
사용자가 실제로 반응한 기사 데이터를 기반으로 관심사를 추정하고, 그 관심사에 가까운 기사를 우선 정렬해 보여주는 개인화 기능입니다.

이 기능의 목적은 다음과 같습니다.

- 사용자가 직접 검색하지 않아도 관심 있는 뉴스를 먼저 보여준다.
- 반복적으로 보는 주제와 키워드를 빠르게 반영한다.
- 이미 본 기사나 덜 관련 있는 기사보다, 지금 읽을 가능성이 높은 기사를 앞쪽에 둔다.
- 로그인 기능 없이도 개인화 경험을 제공한다.

### 2.2 동작 방식

개인화는 서버 계정이 아니라 브라우저 로컬 저장소를 기반으로 동작합니다.

기본 흐름:

1. 사용자가 홈, 뉴스 목록, 뉴스 상세에서 기사를 클릭한다.
2. 클릭한 기사에서 카테고리와 주요 키워드를 추출한다.
3. 해당 카테고리와 키워드의 관심 점수를 올린다.
4. 이미 본 기사 목록에 해당 기사 ID를 저장한다.
5. 이후 맞춤 뉴스 목록을 만들 때 누적 점수를 기준으로 기사별 개인화 점수를 계산한다.
6. 개인화 점수가 높은 기사부터 다시 정렬해 노출한다.

즉, 사용자가 많이 반응한 주제를 시스템이 익명 프로필로 기억하고, 다음 기사 정렬에 반영하는 구조입니다.

### 2.3 어떤 데이터를 반영하는가

현재 맞춤 뉴스는 아래 데이터를 사용합니다.

- 사용자가 클릭한 기사 카테고리
- 사용자가 클릭한 기사 제목 키워드
- 사용자가 클릭한 기사 설명 키워드
- 사용자가 이미 본 기사 목록
- 기사의 최신성

이 데이터는 모두 브라우저에 저장되며, 로그인 계정이나 별도 회원 정보는 사용하지 않습니다.

### 2.4 저장되는 개인화 데이터

브라우저 `localStorage`에 저장되는 주요 데이터는 다음과 같습니다.

- 익명 사용자 ID
- 카테고리별 관심 점수
- 키워드별 관심 점수
- 사용자가 직접 선택한 선호 카테고리
- 사용자가 직접 선택한 선호 키워드
- 이미 읽은 기사 ID 목록
- 생성 시각
- 마지막 갱신 시각

이 구조 덕분에 사용자는 로그인하지 않아도 같은 브라우저 안에서는 개인화 결과를 이어서 볼 수 있습니다.

### 2.5 점수 계산 방식

맞춤 뉴스 정렬은 기사마다 개인화 점수를 계산한 뒤 높은 순서대로 정렬하는 방식입니다.

기본 계산식:

```text
personalizedScore = categoryAffinity * 2 + keywordAffinity + freshness - seenPenalty
```

구성 요소:

- `categoryAffinity`: 사용자가 자주 반응한 카테고리와 얼마나 가까운지
- `keywordAffinity`: 사용자가 자주 반응한 키워드와 얼마나 겹치는지
- `freshness`: 최신 기사인지
- `seenPenalty`: 이미 읽은 기사인지

의미:

- 카테고리 선호는 큰 방향성을 잡는 기준이므로 가중치를 더 크게 둡니다.
- 키워드 선호는 세부 관심사 반영에 사용합니다.
- 최신 뉴스가 너무 뒤로 밀리지 않도록 최신성 가산점을 둡니다.
- 이미 본 뉴스는 반복 노출을 줄이기 위해 감점합니다.

### 2.6 사용자에게 보이는 결과

사용자는 맞춤 뉴스에서 다음과 같은 결과를 체감하게 됩니다.

- 자주 읽는 카테고리 기사가 상단에 더 자주 노출됨
- 관심 키워드가 비슷한 기사가 먼저 보임
- 예전에 읽은 기사보다 새롭고 관련도 높은 기사가 앞쪽에 배치됨
- 홈 화면과 맞춤 탭에서 시간이 지날수록 기사 배열이 개인별로 달라짐

### 2.7 홈 화면과의 연결

맞춤 뉴스는 독립된 화면 기능이면서 동시에 홈 화면의 핵심 축입니다.

적용 위치:

- 홈 상단 탭의 `맞춤`
- 홈 메인 구성 안의 `For You` 영역
- 마이페이지의 관심 카테고리/키워드 요약

즉, 맞춤 뉴스는 단순히 별도 탭 하나가 아니라 서비스 전반의 정렬 기준 역할을 합니다.

### 2.8 마이페이지와의 연결

마이페이지는 맞춤 뉴스의 내부 상태를 사용자가 이해할 수 있게 보여주는 보조 화면입니다.

제공 정보:

- 내가 많이 반응한 카테고리
- 내가 많이 반응한 키워드
- 현재 프로필 기준 선호 흐름

이 화면은 추천 결과를 설명하는 역할을 합니다.  
사용자는 마이페이지를 통해 “왜 이런 뉴스가 맞춤으로 보이는지”를 간접적으로 이해할 수 있습니다.

### 2.9 장점

이 구조의 장점은 다음과 같습니다.

- 로그인 없이 바로 개인화 제공 가능
- 백엔드에 사용자 계정 테이블이 없어도 동작
- 구현 복잡도 대비 체감 효과가 큼
- 기사 클릭 몇 번만으로도 추천 결과 변화가 빠르게 나타남
- 개인정보 저장 부담이 상대적으로 적음

### 2.10 한계

현재 구조의 한계도 명확합니다.

- 브라우저를 바꾸면 개인화 데이터가 이어지지 않음
- 기기를 바꾸면 같은 사용자로 인식할 수 없음
- 클릭 중심 반응만 사용하므로 정교한 체류 시간 분석은 없음
- 서버 계정 기반 추천보다 장기 축적에는 한계가 있음

즉, 현재 맞춤 뉴스는 “가볍고 빠르게 반응하는 개인화”에 최적화된 구조입니다.

### 2.11 향후 확장 방향

맞춤 뉴스는 앞으로 아래 방향으로 더 강화할 수 있습니다.

- 상세 페이지 체류 시간 반영
- 기사 공유/스크롤 깊이 반영
- 관심 없는 기사 숨김
- 추천 이유 문구 제공
- 카테고리/키워드 가중치 조절 UI 제공
- 계정 기반 동기화 기능 추가

## 3. 사용자 화면 기능

### 3.1 홈 화면

홈 화면은 서비스의 메인 진입점이며, 상단 탭 전환으로 여러 섹션을 오갈 수 있습니다.

제공 기능:

- 주요 헤드라인 기사 노출
- 기사 제목/설명 기반 트렌딩 키워드 집계
- 최근 사용 반응 기반 맞춤 뉴스 정렬
- 날씨, 헤드라인, 트렌딩, 맞춤 탭 전환
- 대표 기사 히어로 영역과 요약형 카드 배치

### 3.2 뉴스 목록 화면

뉴스 목록 화면은 기사 탐색 중심 화면입니다.

제공 기능:

- 카테고리별 뉴스 목록 조회
- 검색어 기반 뉴스 검색
- 무한 스크롤 기반 기사 탐색
- 썸네일, 제목, 설명, 출처, 발행 시간 표시
- 카테고리 탭을 통한 빠른 필터링

### 3.3 뉴스 상세 화면

뉴스 상세 화면은 개별 기사 소비를 위한 화면입니다.

제공 기능:

- 기사 제목, 출처, 발행일 표시
- 기사 썸네일 및 본문 렌더링
- 외부 기사 원문 이동
- 상세 진입 시 개인화 반응 데이터 축적

### 3.4 마이페이지

마이페이지는 로그인 없이도 개인화 상태를 확인할 수 있는 화면입니다.

제공 기능:

- 관심 카테고리 확인
- 관심 키워드 확인
- 최근 반응 기반 개인화 상태 확인
- 사용자의 선호 패턴 점검

### 3.5 날씨 화면

날씨 화면은 현재 위치 기반 생활 정보 화면입니다.

제공 기능:

- 현재 위치 기반 날씨 조회
- 시간별 날씨 정보 제공
- 주간 날씨 정보 제공
- 미세먼지 및 대기질 정보 제공
- 위치 권한이 없으면 기본 위치로 대체

## 4. 트렌딩 기능

트렌딩은 기사 제목과 설명에서 반복적으로 등장하는 단어를 기준으로 집계합니다.

제공 방식:

- 최신 기사 목록에서 키워드 추출
- 중복 키워드 빈도 계산
- 상위 키워드를 기준으로 관련 기사 재정렬
- 홈과 트렌딩 탭에서 현재 이슈 흐름 제공

## 5. 백엔드 기능

백엔드는 뉴스 수집, 기사 저장, 기사 조회, 날씨 API 연동을 담당합니다.

### 5.1 뉴스 수집

- Naver 뉴스 검색 API 기반 기사 수집
- 카테고리별 검색어를 사용한 기사 조회
- 정기 배치 실행으로 기사 데이터 갱신
- 시작 시 초기 뉴스 수집 여부를 환경변수로 제어

### 5.2 기사 보강

- 기사 원문 페이지 메타데이터 크롤링
- 제목, 설명, 본문, 이미지, 출처 보강
- HTML 본문을 앱에서 렌더링 가능한 형태로 정리

### 5.3 데이터 저장

- Neon PostgreSQL + Prisma 사용
- 기사, 카테고리, 날씨 캐시 데이터 저장
- 중복 URL 기준 upsert 처리

### 5.4 날씨 연동

- Open-Meteo API 연동
- 현재 날씨, 시간별 예보, 주간 예보 조회
- 공기질 API와 결합해 미세먼지 정보 제공
- 캐시 저장으로 동일 좌표 요청 최적화

### 5.5 운영용 API

- `/health` 헬스체크 제공
- `/api` Swagger 문서 노출 여부를 환경변수로 제어
- CORS 허용 대상을 환경변수로 제어

## 6. 배치와 운영 설정

운영 시 중요한 백엔드 설정은 다음과 같습니다.

- `ENABLE_SWAGGER`
- `ENABLE_NEWS_SCHEDULE`
- `ENABLE_STARTUP_NEWS_FETCH`
- `CORS_ORIGIN`
- `DATABASE_URL`

운영 권장값:

- Swagger 비활성화
- 시작 시 전체 뉴스 강제 수집 비활성화
- 정기 뉴스 배치는 활성화
- CORS는 실제 프론트 도메인만 허용

## 7. 현재 문서 기준 운영 범위

현재 프로젝트는 `로컬`, `운영` 두 환경만 관리합니다.

- 로컬: 개발 및 확인용
- 운영: 실제 배포용

## 8. 핵심 사용자 흐름

대표적인 사용자 흐름은 아래와 같습니다.

1. 홈 진입
2. 헤드라인 기사 확인
3. 뉴스 목록 또는 상세 진입
4. 클릭 이력이 익명 프로필에 반영
5. 맞춤 탭 정렬 결과 변경
6. 재방문 시 관심사 기반 기사 우선 노출

## 9. 추후 확장 후보

- 기사 공유 기능
- 즐겨찾기 기능
- 읽은 기사 숨김 기능
- 추천 이유 문구 강화
- 배치 모니터링 및 실패 알림
